## 问题定义

- 当前问题:
  - 当前 WebUI 前端完全以内嵌 HTML/CSS/JS 字符串形式存在于 `webui/html.go` 与 `webui/dashboard.go`。
  - 登录页与仪表盘都采用高饱和黑绿 CRT/终端风格，颜色和阴影大量硬编码，缺少统一主题抽象。
  - 现有前端只有语言切换，`localStorage` 里只保存 `lang`，没有主题切换、浅色模式或护眼配色能力。
- 影响对象:
  - 访客与管理员两类 WebUI 用户
  - 登录页、首页仪表盘、侧边栏、弹窗、表格、按钮、日志面板等全部前端界面
- 触发背景:
  - 用户希望当前项目增加一套前端主题界面管理系统，提供浅色和深色的多套主题风格，并强调“简洁、大气、便于阅读、护眼”。
- 为什么现在处理:
  - 当前视觉风格辨识度强，但对长时间阅读和普通办公环境不够友好。
  - 现有前端仍是单体嵌入式实现，趁主题相关变量还集中在 `:root` 与硬编码样式里，适合先建立主题基础层，避免后续继续堆叠颜色常量。

## 目标与成功标准

- 目标 1:
  - 为 WebUI 建立统一的前端主题管理机制，而不是继续在多个 HTML 字符串里分散硬编码颜色。
- 目标 2:
  - 提供至少 4 套内置主题：不少于 2 套浅色主题、2 套深色主题。
- 目标 3:
  - 主题覆盖登录页与仪表盘主界面，保证视觉语言统一。
- 目标 4:
  - 默认主题整体变得更易读、更护眼，降低当前高饱和荧光绿带来的视觉疲劳。
- 目标 5:
  - 主题选择可在前端持久化，刷新页面后仍保持。

## 范围边界

### 本次必须完成

- 设计并落地一套前端主题 token / 变量体系。
- 给登录页与仪表盘接入统一主题切换逻辑。
- 实现浅色 / 深色多主题切换入口与本地持久化。
- 至少完成 4 套主题预设，并覆盖核心组件：
  - 页面背景
  - 卡片/面板
  - 表格
  - 按钮
  - 输入框
  - 弹窗
  - 标签/状态色
- 保持现有 i18n、登录逻辑、配置保存与 API 调用链不被破坏。

### 本次明确不做

- 不引入独立前端工程、Node 构建链或外部 UI 框架。
- 不改动后端 API、数据库 schema、登录权限模型。
- 不把主题选择持久化到服务端用户配置；本轮默认使用浏览器本地持久化。
- 不顺手重做整套信息架构或交互流程，只聚焦主题系统与视觉一致性。

## 验收真相陈述

- [REQ-001] 登录页与仪表盘必须共享统一的主题管理机制，不能继续各自维护独立的硬编码配色体系。
- [REQ-002] 系统必须提供不少于 4 套内置主题，且同时包含浅色与深色主题。
- [REQ-003] 用户切换主题后，刷新页面仍能恢复上次选择。
- [REQ-004] 新主题默认风格需要满足“简洁、大气、便于阅读、护眼”，文本对比和背景亮度不能明显退化。
- [REQ-005] 现有语言切换、管理员/访客权限显示、配置弹窗与数据加载链路不能因主题系统失效。

## 约束与依赖

- 技术约束:
  - 当前前端不是独立工程，而是 `webui/html.go` 与 `webui/dashboard.go` 中的内嵌 HTML/CSS/JS。
  - 当前样式变量只在 `dashboard.go` 的 `:root` 中局部存在，登录页仍是纯硬编码颜色。
  - 当前浏览器本地状态只使用 `localStorage.lang`，主题持久化最好沿用同一机制。
- 业务约束:
  - 管理面板是项目核心使用入口，不能为了主题改造破坏可读性和操作效率。
  - 登录页和仪表盘都需要保持统一品牌识别，但不能再走过强的“荧光终端”刺激风格。
- 外部依赖:
  - 无新增后端依赖。
  - 如果需要运行态验证，依赖本地 Go 运行环境或可访问的 WebUI 实例；当前会话尚未验证这类环境可用。

## 风险与待确认项

- 风险:
  - 现有样式高度耦合，若只局部替换颜色变量，容易遗漏 hover、阴影、边框、状态色等细节。
  - 登录页与仪表盘分散在不同文件里，若不抽公共 token，会再次产生视觉漂移。
  - 当前没有现成前端测试框架，运行态验证主要依赖页面实际加载与代码级静态检查。
- 待确认:
  - 主题切换入口最终放在顶部导航、设置面板还是两者同时提供；本轮 discovery 默认推荐放在顶部导航，保证访客和管理员都能切换。
  - 用户没有指定具体主题数量和命名，本轮按“至少 4 套（2 浅 + 2 深）”作为可执行基线假设推进。
