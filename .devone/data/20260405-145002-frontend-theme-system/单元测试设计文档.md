## 测试目标

- 需要被保护的核心行为:
  - 主题选择能作用到登录页与仪表盘两类页面。
  - 主题切换后刷新页面能恢复同一主题。
  - 主题切换不会破坏现有语言切换与页面主要操作。
- 这轮必须覆盖的失败模式:
  - 主题 key 无效时页面样式崩坏
  - 登录页和仪表盘主题不一致
  - 切换主题后 `localStorage` 未保存或恢复失败
  - 某套主题缺少关键 token，导致按钮/表格/弹窗出现不可读状态

## 单元测试设计

### UT-001

- 目标:
  - 校验主题预设定义完整，所有主题都包含同一组关键 token。
- 覆盖函数/模块:
  - `webui/theme.go` 中的主题 token 注册表
- 输入:
  - 每个内置主题的 CSS 变量集合
- 预期输出:
  - 每套主题都具备完整字段：背景、卡片、文本、次文本、边框、强调色、成功/警告/危险色等
- 失败表现:
  - 某个主题缺字段，应用后组件出现空白色值或视觉断层

### UT-002

- 目标:
  - 校验主题恢复逻辑：无保存值、合法保存值、非法保存值三种路径。
- 覆盖函数/模块:
  - `webui/theme.go` 中的 `resolveThemeKey` / `detectDefaultTheme` / `loadThemePreference`
- 输入:
  - `localStorage['goproxy.theme'] = null | validKey | invalidKey`
- 预期输出:
  - `null` -> 按系统偏好回落到 `mist-light` 或 `graphite-dark`
  - `validKey` -> 对应主题
  - `invalidKey` -> 明确回退 `graphite-dark`
- 失败表现:
  - 页面刷新后无法恢复、报错或落入未定义主题

## 属性测试设计

### PT-001

- 性质:
  - 任意内置主题切换后，页面必须始终保持“可读 token 完整性”。
- 输入空间:
  - 所有内置主题 key 的任意切换顺序
- 不变量:
  - 主题切换后仍能找到完整 token 集
  - 语言切换状态与主题状态可并存
  - 不会移除核心 DOM 或破坏 API 加载逻辑
- 预期失败模式:
  - 多次切换后部分 token 丢失、标题按钮颜色失真、i18n 或表格刷新异常
