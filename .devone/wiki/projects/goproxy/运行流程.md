# 运行流程

GoProxy 的运行流程可以分成 4 条主线：启动装配、免费代理补池、订阅刷新 / 唤醒、对外请求转发。理解这 4 条主线后，再读单个模块会轻松很多。

## 1. 启动装配流程

1. `logger.Init()` 启动内存日志收集。
2. `config.Load()` 合并环境变量和 `config.json`。
3. `storage.New(cfg.DBPath)` 打开 SQLite 并执行 schema 初始化 / 迁移。
4. 初始化 `SourceManager`、`Fetcher`、`Validator`、`pool.Manager`、`HealthChecker`、`Optimizer`、`custom.Manager`。
5. 创建 `webui.Server`，并把 `smartFetchAndFill` 作为回调传给 WebUI。
6. 启动：
   - 初次补池 goroutine
   - 状态监控
   - 健康检查
   - 优化轮换
   - 订阅管理
   - 配置变更监听
   - HTTP / SOCKS5 四个服务端口

## 2. 免费代理补池流程

1. `startStatusMonitor()` 每 30 秒检查一次池状态。
2. `poolMgr.NeedsFetch()` 决定是否需要抓取，以及使用 `emergency` / `refill` / `optimize` 哪种模式。
3. `fetcher.FetchSmart()` 选择源：
   - `emergency`：所有源，忽略断路器
   - `refill`：快更新源
   - `optimize`：随机少量慢更新源
4. `validator.ValidateStream()` 并发验证代理。
5. `smartFetchAndFill()` 过滤无出口信息、超延迟、地理不合规节点。
6. `poolMgr.TryAddProxy()` 决定直接入池、浮动入池或替换慢节点。

## 3. 订阅刷新与唤醒流程

### 刷新流程

1. `custom.Manager.Start()` 启动后立即执行 `initialRefresh()`。
2. `refreshLoop()` 每分钟检查一次所有活跃订阅是否到刷新时间。
3. `RefreshSubscription()` 拉取 URL 或读取本地文件，再调用 `Parse(data, sub.Format)`。
4. 节点分为两类：
   - 直连节点：直接入池
   - 隧道节点：收集后交给 `sing-box.Reload()`，再把生成的本地 `127.0.0.1:{port}` SOCKS5 地址入池
5. `validateCustomProxies()` 验证这些订阅代理：
   - 有效：更新出口信息并启用
   - 无效：禁用
6. 若某订阅存在可用节点，则更新 `subscriptions.last_success`。

### 唤醒流程

1. `probeLoop()` 按 `CustomProbeInterval` 定时扫描被禁用的订阅代理。
2. 对禁用代理再次做 `ValidateOne()`。
3. 验证通过且不触发地理过滤则重新启用，否则继续保持禁用。

### 清理流程

1. `cleanupStaleSubscriptions()` 查询连续 7 天无可用节点的活跃订阅。
2. 删除其关联代理与订阅记录。
3. 若发生删除，再执行一次 `RefreshAll()` 重建 `sing-box` 配置。

## 4. 对外请求转发流程

### HTTP

1. `proxy.Server.ServeHTTP()` 先做可选的 Basic Auth。
2. 若是 `CONNECT` 请求走 `handleTunnel()`；普通请求走 `handleHTTP()`。
3. `selectProxy()` 根据端口模式与 `custom_proxy_mode` 选择随机或最低延迟节点。
4. 转发失败时：
   - 免费代理：删除
   - 订阅代理：禁用
5. 成功时记录使用统计与日志。

### SOCKS5

1. `SOCKS5Server.Start()` 接受 TCP 连接。
2. `socks5Handshake()` 处理无认证或用户名 / 密码认证。
3. `selectSOCKS5Proxy()` 只从 SOCKS5 协议上游节点中选择。
4. 建立上游连接后双向 `io.Copy()` 转发数据。

## 5. 配置变更流程

1. WebUI `POST /api/config/save` 解析请求体。
2. `config.Save()` 持久化到 `config.json` 并更新内存配置。
3. 通过 `configChanged` channel 通知主进程。
4. `watchConfigChanges()` 检测到容量或比例变更后调用 `poolMgr.AdjustForConfigChange()`。

## 后台循环总表

| 流程 | 触发位置 | 默认间隔 | 作用 |
| --- | --- | --- | --- |
| 状态监控 | `main.go` | 30 秒 | 发现池子不足并触发补池 |
| 健康检查 | `checker/health_checker.go` | `HealthCheckInterval` 分钟，默认 5 | 批量重验代理 |
| 优化轮换 | `optimizer/optimizer.go` | `OptimizeInterval` 分钟，默认 30 | 用更快代理替换慢节点 |
| 订阅刷新检查 | `custom/manager.go` | 1 分钟 tick | 到期订阅刷新 |
| 禁用节点探测 | `custom/manager.go` | `CustomProbeInterval` 分钟，默认 10 | 唤醒被禁用订阅节点 |

## 已确认 / 推断 / 未知

### 已确认

- 只有 HTTP 隧道与 SOCKS5 代理转发链路会直接删除 / 禁用失败节点。
- 订阅节点刷新完成后并不是立即全部可用，而是要经过 `validateCustomProxies()` 二次验证。

### 推断

- 当前免费代理补池和订阅刷新都可能并发触发，依赖各自局部锁与数据库去重来避免重复入池。

### 未知

- 仓库内没有端到端请求时序图，复杂故障下的跨 goroutine 顺序只能从日志恢复。
