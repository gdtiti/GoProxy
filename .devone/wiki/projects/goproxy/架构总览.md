# 架构总览

GoProxy 的真实形态是一个单进程 Go 服务，进程内同时承担抓取、验证、池化、WebUI 和对外代理转发；只有订阅中的加密节点会额外借助 `sing-box` 子进程转换为本地 SOCKS5 入口。

## 系统形态

- 系统类型：单体 Go 守护进程
- 主要运行单元：
  - 主进程：`main.go` 装配全部模块
  - 子进程：`custom/singbox.go` 启动和重载 `sing-box`
- 核心依赖：
  - SQLite：代理、订阅、源状态、运行时配置
  - 外部 HTTP 源：公开代理列表、`ip-api.com`
  - Docker：容器化部署

## 组件边界

### 启动与编排：`main.go`

- 作用：初始化日志、配置、存储、抓取器、验证器、池管理器、后台任务、WebUI 与 4 个代理端口。
- 关键文件：`main.go`
- 与其他组件的关系：是所有组件的唯一装配点。

### 配置层：`config/`

- 作用：从环境变量和 `config.json` 生成全局配置，并通过 `sync.RWMutex` 暴露线程安全访问。
- 关键文件：`config/config.go`
- 与其他组件的关系：所有模块通过 `config.Load()` / `config.Get()` 读取。

### 存储层：`storage/`

- 作用：管理 SQLite 连接、建表与迁移，并提供代理、订阅、源状态等数据访问方法。
- 关键文件：`storage/storage.go`
- 与其他组件的关系：`fetcher`、`pool`、`checker`、`custom`、`webui`、`proxy` 都依赖它。

### 抓取与断路器：`fetcher/`

- 作用：从多组公开源并发抓取代理，并把成功 / 失败记录到 `source_status` 断路器状态表。
- 关键文件：`fetcher/fetcher.go`、`fetcher/source_manager.go`
- 与其他组件的关系：`main.go` 与 `optimizer` 都通过它拿候选代理。

### 验证层：`validator/`

- 作用：验证代理连通性、响应时间、出口 IP / 地理位置，以及 HTTP 代理的 HTTPS CONNECT 能力。
- 关键文件：`validator/validator.go`
- 与其他组件的关系：免费代理入池、健康检查、订阅代理验证都依赖它。

### 池管理：`pool/`

- 作用：根据容量、协议比例和延迟阈值判定池状态，并执行入池、浮动入池或替换逻辑。
- 关键文件：`pool/manager.go`
- 与其他组件的关系：`main.go`、`optimizer`、`webui` 都依赖其状态与调整逻辑。

### 后台维护：`checker/` + `optimizer/`

- 作用：
  - `checker/health_checker.go`：批量健康检查与清理
  - `optimizer/optimizer.go`：健康状态下抓取更优代理并替换慢节点
- 与其他组件的关系：都依赖 `storage`、`validator`、`pool`

### 订阅管理：`custom/`

- 作用：拉取订阅、解析节点、区分直连节点与需隧道节点、驱动 `sing-box`、并维护刷新 / 探测唤醒循环。
- 关键文件：`custom/manager.go`、`custom/parser.go`、`custom/singbox.go`
- 与其他组件的关系：被 `webui` 调用，也在 `main.go` 启动后台循环。

### 对外代理服务：`proxy/`

- 作用：暴露 HTTP 与 SOCKS5 服务，按模式选择上游代理并负责失败重试。
- 关键文件：`proxy/server.go`、`proxy/socks5_server.go`
- 与其他组件的关系：直接读取 `storage` 中的可用节点。

### 管理面板：`webui/`

- 作用：暴露登录页、仪表盘 HTML、只读与管理员 API，并可触发配置保存、抓取和订阅刷新。
- 关键文件：`webui/server.go`、`webui/html.go`
- 与其他组件的关系：调用 `storage`、`poolMgr`、`customMgr`，并通过 channel 通知配置变更。

## 关键调用链

### 启动链

1. `main.go` 调用 `config.Load()` 生成全局配置。
2. `storage.New()` 打开 SQLite 并自动建表 / 迁移。
3. 主进程创建 `fetcher`、`validator`、`pool.Manager`、`HealthChecker`、`Optimizer`、`custom.Manager`。
4. `webui.New(...).Start()` 先启动 WebUI，再并发启动状态监控、健康检查、优化轮换、订阅管理和 4 个代理端口。

### 免费代理链

1. `smartFetchAndFill()` 调用 `poolMgr.GetStatus()` 与 `NeedsFetch()` 判定是否缺代理。
2. `fetcher.FetchSmart()` 按模式挑选源并抓取候选。
3. `validator.ValidateStream()` 并发验证连通性、地理位置与延迟。
4. `poolMgr.TryAddProxy()` 决定直接入池、浮动入池、替换或拒绝。
5. `storage` 落库后，由健康检查和优化器持续维护。

### 订阅链

1. WebUI API 接收 URL 或文件，先用 `customMgr.ValidateSubscription()` 做静态验证。
2. `customMgr.RefreshSubscription()` 拉取订阅、解析节点、删除旧代理。
3. 直连节点直接 `AddProxyWithSource(..., "custom")`；加密节点经 `sing-box.Reload()` 生成本地 SOCKS5 端口后再入池。
4. `validateCustomProxies()` 逐个验证并按结果启用 / 禁用，同时更新 `last_success`。

### 出口代理链

1. HTTP / SOCKS5 服务根据模式和 `custom_proxy_mode` 选择上游节点。
2. 请求转发失败时，免费代理删除、订阅代理禁用。
3. 成功请求记录使用统计，供后续替换和健康检查使用。

## 外部依赖

- 数据库：SQLite（无独立缓存、无消息队列）
- 第三方 HTTP 服务：
  - 多个 GitHub / jsDelivr 代理列表源
  - `ip-api.com` 用于出口 IP 和地理位置查询
- 子进程：`sing-box`
- 部署：Docker / Docker Compose

## 设计约束

- SQLite 连接显式设置 `MaxOpenConns(1)`，默认按单写场景设计。
- 订阅代理不受免费池 slot 容量限制；免费代理必须经过协议槽位控制。
- HTTP 上游代理必须额外通过 HTTPS CONNECT 检查。
- 地理过滤优先使用白名单；白名单为空时才使用黑名单。

## 已确认 / 推断 / 未知

### 已确认

- `HealthChecker` 与 `Optimizer` 都使用定时器后台运行。
- `SourceManager` 会把抓取源降级或禁用，并在冷却结束后自动恢复为可用。
- WebUI 会把 `config.Save()` 的结果通过 channel 通知给主进程，再由 `poolMgr.AdjustForConfigChange()` 处理容量变化。

### 推断

- 当前进程没有显式优雅退出编排；若要补充 shutdown，需要同时处理 WebUI、4 个代理端口、ticker 与 `sing-box` 子进程。

### 未知

- 镜像发布与版本同步策略不在仓库内，无法从源码判断镜像标签与代码版本是否一一对应。
