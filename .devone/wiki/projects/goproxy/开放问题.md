# 开放问题

本页只保留“当前仓库里已经看到矛盾或风险，但仍有边界未闭合”的事项。和上一轮相比，有几项已经从“模糊怀疑”收敛为“已确认的一致性问题 + 尚未解释的原因”。

## 1. `checker/` 目录内确实并存两套健康检查实现，但旧实现已确认不在当前主链

- 已确认：
  - `main.go` 实际装配的是 `checker.NewHealthChecker(...)`，并启动 `healthChecker.StartBackground()`。
  - `checker/checker.go` 中旧 `Checker.New()` / `Start()` 没有进入当前 `main.go` 主调用链。
- 仍待确认：
  - 旧实现是否只是历史残留，还是仍被仓库外脚本 / 其他分支依赖。
- 结论状态：**主链未使用** 已确认；**保留原因** 未知。

## 2. 测试文档与当前仓库产物命名确实不一致

- 已确认：
  - `test/README.md` 引用了不存在的 `TEST_GUIDE.md`。
  - `test/README.md` 写的是启动 `./goproxy`。
  - 主 README 写的是 `go build -o proxygo . && ./proxygo`。
  - Dockerfile / 容器文档体系又使用了 `proxy-pool` / 预构建镜像路径。
- 结论状态：**文档不一致** 已确认；**应以哪个名字为准** 仍需维护者明确。

## 3. 仓库远端与文档 / 镜像发布命名存在明显分叉

- 已确认：
  - Git 远端是 `https://github.com/gdtiti/GoProxy.git`
  - README、CHANGELOG、`docker-compose.yml` 多处仍指向 `isboyjc/GoProxy` 与 `ghcr.io/isboyjc/goproxy`
- 当前更准确的判断：
  - 这不是单一文件的笔误，而是**仓库归属 / 发布命名长期分叉**。
- 仍待确认：
  - 这是 fork、迁移、镜像发布者变更，还是文档未同步，当前仓库内没有直接说明。

## 4. 根目录 `proxygo` 不是临时文件，而是已被 git 跟踪的二进制

- 已确认：
  - `proxygo` 存在于仓库根目录。
  - `git ls-files --stage -- proxygo` 显示它是已跟踪文件。
  - `git log --oneline --follow -- proxygo` 只追到初始提交 `f55209d feat: ✨ init`。
- 仍待确认：
  - 为什么要把编译产物纳入版本控制，仓库内没有解释。

## 5. WebUI 登录会话仅保存在进程内存

- `sessions` 是包级 map，重启进程后会话全部失效。
- 这是合理的简化方案，但仓库内没有更明确的安全 / 运维说明。

## 已确认 / 推断 / 未知

### 已确认

- 第 1 项里“旧 `Checker` 不在当前主链”已经确认。
- 第 2、3、4 项都不是猜测，而是当前仓库可以直接验证的一致性问题。
- 第 5 项的会话存储方式也能从源码直接确认。

### 推断

- 第 2、3、4 项更像仓库维护一致性问题，而不是运行时 bug。
- 第 3 项更像 fork / 迁移 / 发布命名分叉留下的文档痕迹。

### 未知

- 第 1 项旧实现的保留目的。
- 第 2 项应统一到哪个最终产物名。
- 第 3、4 项背后的维护决策。
- 维护者是否已经在仓库外部文档或发布平台解释这些差异，当前仓库内没有证据。
